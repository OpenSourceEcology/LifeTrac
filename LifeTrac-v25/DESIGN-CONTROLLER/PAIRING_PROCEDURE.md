# LifeTrac v25 Pairing Procedure

**Status:** DRAFT FOR REVIEW. Operator-facing procedure for binding a
handheld to a base/tractor pair. Distinct from key *rotation* (covered
in [KEY_ROTATION.md](KEY_ROTATION.md)) — pairing is the act of teaching a
*newly-provisioned* radio that a specific tractor is its peer.

**Companions:** [KEY_ROTATION.md](KEY_ROTATION.md), [SAFETY_CASE.md](SAFETY_CASE.md),
[CYBERSECURITY_CASE.md](CYBERSECURITY_CASE.md).

## 1. When to pair

| Trigger | Action |
|---------|--------|
| New handheld out of the box | Initial pair (§3) |
| New tractor at the install bench | Initial provision (§2) |
| Lost handheld → replacing | Initial pair (§3) **plus** rotate fleet key (KEY_ROTATION.md §4) |
| Operator suspects compromise | Rotate fleet key first, then re-pair every node |
| Periodic (annual) | Rotate fleet key; re-pair only if keying tool failed on any node |

## 2. Initial provisioning (install bench)

Before a tractor or handheld can pair, it needs:

1. The current fleet key (32 bytes, generated by `tools/keygen.py`).
2. Its own `source_id` byte (`SRC_HANDHELD`, `SRC_BASE`, `SRC_AUTONOMY`).
3. A unique `node_id` (16-bit) so audit logs distinguish two handhelds.

```bash
# At the install bench, with the device on USB:
python tools/keygen.py > keys/fleet.hex     # one time, per fleet
python tools/provision.py \
    --write-port /dev/ttyACM0 \
    --key keys/fleet.hex \
    --source-id 0x01 \
    --node-id 0x0042
```

The provisioning tool writes to a dedicated flash region the firmware
reads at boot. See `tools/provision.py` for the wire format.

## 3. Field pairing (no install bench)

When swapping a handheld in the field (e.g. one was dropped):

1. Tech walks to the tractor with the **new** handheld.
2. Operator power-cycles both the tractor and the new handheld.
3. With the tractor running, hold the handheld within 5 m of the tractor
   antenna and press the **PAIR** button on the handheld for 3 s.
4. The handheld broadcasts a `CMD_PAIR_REQUEST` (FT_COMMAND opcode 0x70)
   carrying its `source_id` and `node_id`.
5. The tractor M7, if a tech has armed pairing mode on the operator UI
   (`/api/pair/arm` POST, requires PIN), responds with `CMD_PAIR_ACK`
   carrying the current `fleet_key_id` echo for confirmation.
6. The handheld latches the peer relationship and beeps twice. Done.

Pairing mode auto-disarms after 60 s. The tractor refuses to pair when:

- the operator UI has not armed pairing,
- the candidate handheld's GCM tag does not validate (already-keyed
  attacker cannot brute-force in),
- a different handheld with the same `source_id` is already streaming
  ControlFrames in the same minute (anti-collision).

## 4. Unpair / decommission

To remove a handheld (sold, lost, retired):

```bash
# At the install bench:
python tools/provision.py --erase --write-port /dev/ttyACM0
# AND on every other node in the fleet:
python tools/keygen.py > keys/fleet.hex     # new key
# Re-provision every remaining node with the new key.
```

Erasing the lost handheld's flash is best-effort (we don't physically
have it); rotating the fleet key is the actual mitigation. See
[KEY_ROTATION.md](KEY_ROTATION.md).

## 5. Audit

Every pair / unpair / arm / disarm event lands in `audit_log.jsonl` with
event `pair_*`. The base operator UI at `/audit` shows them with the
`pair` highlight.

## 6. Recovery from a bad pair

Symptom: handheld LED is green, but the tractor refuses commands and the
audit log shows repeated `gcm_reject` from the handheld's `source_id`.

Diagnosis: the handheld has a stale fleet key.

Fix: bring the handheld to the install bench and re-provision per §2.
Field re-key is **not** supported in v25 — the fleet key must be loaded
over wired USB.
