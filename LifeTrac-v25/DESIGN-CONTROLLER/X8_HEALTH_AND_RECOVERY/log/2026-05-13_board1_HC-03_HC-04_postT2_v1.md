# 2026-05-13 — Board 1 (`2D0A`) HC-03 + HC-04 post-T2 — SWD wedge persists

**Routines:** [HC-03_h7_swd_attach_diff.md](../routines/HC-03_h7_swd_attach_diff.md), [HC-04_stage1_standard_quant.md](../routines/HC-04_stage1_standard_quant.md)

**Context:** Earlier today the user performed a cold power cycle on Board 1
(see [2026-05-13_board1_T2_postcycle_v1.md](2026-05-13_board1_T2_postcycle_v1.md)).
HC-01 + HC-02 came back PASS. The next step in the playbook was to confirm
HC-03 + HC-04 before resuming the W1-10b campaign.

## HC-03 timeline (Board 1)

| Step | `srst` | DPIDR | Notes |
|---|---|---|---|
| First probe (no preflight) | **0** | `cannot read IDR` | gpio10 still in alt-function on fresh boot. |
| After `manual_gpio_preflight.sh` | **1** | `cannot read IDR` | gpio10 cleared. New signature. |
| After `prep_bridge.sh` + preflight | **1** | `cannot read IDR` | x8h7 modules unloaded; no improvement. |

**Diagnosis:** the pad drift has migrated. gpio10 (NRST control) is now
correctly muxed as GPIO output high after preflight, but the SWD pads
(gpio15 = SWCLK, gpio8 = SWDIO) appear to be in alt-function or otherwise
non-responsive. Openocd reads `srst 1` but cannot drive a successful DP
sync.

## Control: HC-04 on Board 2 (`2E2C`)

3 cycles, **3/3 PASS** in 76 s. Run id
`T6_stage1_standard_quant_2026-05-13_095554_724-44412`. Confirms host
pipeline + tooling are healthy; failure on Board 1 is board-specific.

## Conclusion

T2 cold cycle (~10 s hold) cleared the Linux user-space hang but **did
not** clear the SWD-side IOMUXC wedge. Two paths forward:

1. **Longer-hold T2** (e.g. 30–60 s with the carrier completely
   de-energized; possibly remove the coin-cell if present) to give the
   IOMUXC retention domain a chance to reset.
2. **T3a SDP/uuu reflash** (procedure in
   [T3a_sdp_uuu_reflash.md](../recovery/T3a_sdp_uuu_reflash.md)). Even if
   the eMMC image is fine, an SDP boot bypasses Linux entirely and the
   first userland Linux loads after reflash usually starts with a clean
   pinmux.
