# Code Review: LifeTrac v25 Control Stack
**Reviewer:** GPT-5.4  
**Date:** 2026-04-16

## Scope
This review pass focused on the software control path for LifeTrac v25:

- `arduino_opta_controller/lifetrac_v25_controller.ino`
- `esp32_remote_control/lifetrac_v25_remote.ino`
- `raspberry_pi_web_controller/app.py`
- `raspberry_pi_web_controller/static/js/controller.js`
- supporting configuration and README files where behavior and deployment expectations diverge

I did not re-review the full mechanical OpenSCAD system in this pass; earlier January 2026 reviews already covered that area.

## Findings

### 1. Critical: MQTT joystick values are coerced to integers, breaking proportional control
In the Opta MQTT callback, each joystick field is read with `doc["..."] | 0` at [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L329), [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L330), [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L331), and [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L332).

ArduinoJson infers the requested type from the default value, so `| 0` requests `int`, not `float`. All MQTT producers in this repo publish floating-point joystick values in the `-1.0..1.0` range, including the ESP32 remote and the web controller. That means values like `0.3`, `0.5`, or `-0.7` will be truncated before assignment, collapsing MQTT control to effectively `-1`, `0`, or `1`.

Impact:
- Proportional drive and hydraulic control are lost in MQTT mode.
- The README and ROS2/web interfaces all advertise analog control that the controller does not actually honor.

### 2. Critical: Broker loss can leave hydraulic outputs latched because reconnect blocks the safety timeout
In MQTT mode, the main loop calls `reconnectMQTT()` before the timeout check at [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L226) and [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L232). The reconnect routine then blocks in a `while (!client.connected())` loop with 5 second sleeps at [arduino_opta_controller/lifetrac_v25_controller.ino](../../arduino_opta_controller/lifetrac_v25_controller.ino#L289).

If the MQTT connection drops while the machine is moving, the previously energized outputs remain in their last state. `stopAllMovement()` is not called on disconnect, and the safety timeout is not evaluated again until reconnect returns. A network outage can therefore extend motion well past the intended 1 second safety window.

Impact:
- Loss of broker or WiFi can produce uncontrolled continued motion.
- This undermines the primary fail-safe described by `SAFETY_TIMEOUT`.

### 3. Critical: Keyboard control in the web UI keeps sending the last motion command after key release
`updateKeyboardControl()` writes command values when keys are pressed at [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L170), [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L171), [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L172), and [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L173). But when all keys are released, the `activeKeys.size === 0` branch at [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L177) only refreshes the display; it never clears `currentControl` back to zero.

The command loop continues emitting whatever stale values remain every 50 ms via [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L321), [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L322), [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L330), and [raspberry_pi_web_controller/static/js/controller.js](../../raspberry_pi_web_controller/static/js/controller.js#L339).

Reproduction:
1. Press `W` in the browser.
2. Release `W`.
3. The UI keeps transmitting the previous forward command until an emergency stop or some other input overwrites it.

Impact:
- Browser keyboard control can latch motion indefinitely.
- The Opta timeout never trips because the browser continues sending non-zero commands.

### 4. High: The ESP32 remote keeps publishing stale commands if a joystick disconnects
The remote only refreshes joystick axes when each joystick is still connected at [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L178) and [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L192). If a joystick disappears from the I2C bus mid-operation, the corresponding `currentControl` fields are left unchanged, preserving the last non-zero command values from [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L183), [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L184), [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L197), and [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L198).

The remote then keeps transmitting on its periodic path through [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L206), [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L214), [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L216), and [esp32_remote_control/lifetrac_v25_remote.ino](../../esp32_remote_control/lifetrac_v25_remote.ino#L219).

Impact:
- A loose joystick cable or I2C failure can keep drive or hydraulic motion commanded.
- This is another stale-command hazard in a safety-critical path.

### 5. High: The web controller exposes machine control to any origin with no authentication gate
The Flask-SocketIO server is created with `cors_allowed_origins="*"` at [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L41). Control and emergency-stop commands are accepted directly by the socket handlers at [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L243) and [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L272), with no session, token, origin, or client authorization checks.

In practice, any browser tab on the same reachable network can open a Socket.IO connection and issue control commands. Because this system controls heavy hydraulic machinery, the lack of an authorization boundary is not just a general web-security issue; it is an operational safety issue.

### 6. Medium: The documented runtime config file is not actually consumed by the web controller
The web controller README instructs the operator to edit `config/config.yaml` at [raspberry_pi_web_controller/README.md](../../raspberry_pi_web_controller/README.md#L82), [raspberry_pi_web_controller/README.md](../../raspberry_pi_web_controller/README.md#L276), and [raspberry_pi_web_controller/README.md](../../raspberry_pi_web_controller/README.md#L324). The same file is treated as runtime state in [raspberry_pi_web_controller/.gitignore](../../raspberry_pi_web_controller/.gitignore#L17).

However, the actual application hardcodes the broker, password, camera settings, and secret key directly in [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L40), [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L44), [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L47), [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L52), [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L53), and [raspberry_pi_web_controller/app.py](../../raspberry_pi_web_controller/app.py#L54). There is no code path that loads the YAML configuration.

Impact:
- Operators following the documented configuration flow will not affect runtime behavior.
- Deployments that change broker address, credentials, or camera tuning in `config/config.yaml` will silently keep using compiled-in defaults.

## Overall Assessment
The most important issues in this pass are not stylistic; they are stale-command and fail-safe failures in the live control path. The top four findings all allow a last-known motion command to persist longer than intended, or degrade analog control into coarse step control.

## Recommended Fix Order
1. Fix the Opta MQTT float parsing and reconnect fail-safe behavior first.
2. Fix stale-command clearing in both browser keyboard control and the ESP32 remote.
3. Add a real authorization boundary to the web controller before relying on browser-based control outside a tightly isolated test network.
4. Make the web controller actually load `config/config.yaml` so deployment instructions match runtime behavior.
